靜態和動態渲染

靜態渲染 (預設)​

預設情況下，Next.js 會以靜態方式呈現路由以提高性能。這意味著所有的渲染工作都是提前完成的，並且可以從距離用戶地理位置更近的內容交付網絡（CDN）提供服務。

靜態獲取資料 (預設)​

預設情況下，Next.js 會緩存 fetch() 請求的結果，除非明確選擇不進行緩存。這意味著沒有設置緩存選項的 fetch 請求將使用 force-cache 選項。

如果路由中的任何 fetch 請求使用了 revalidate 選項，則在重新驗證期間將靜態地重新渲染該路由。

動態渲染​

在靜態渲染期間，如果發現動態函數或動態 fetch() 請求（無緩存），Next.js 將在請求時切換為動態渲染整個路由。在動態渲染期間，任何已緩存的數據請求仍然可以被重用。

請注意，動態函數無論數據請求是否緩存，都會將路由切換為動態渲染。換句話說，靜態渲染不僅取決於數據請求行為，還取決於路由中使用的動態函數。

動態函式​

動態函數依賴於僅在請求時才能知道的信息，例如用戶的 cookie、當前請求的標頭或 URL 的搜索參數。在 Next.js 中，這些動態函數包括：

  • 在 Server Component 中使用 cookies() 或 headers() 將整個路由切換為動態渲染，僅在請求時才執行。
  • 在 Client Component 中使用 useSearchParams() 將跳過靜態渲染，而是在客戶端上渲染所有 Client Component，直到最近的父 Suspense 邊界。
    • 我們建議將使用 useSearchParams() 的 Client Component 包裹在 <Suspense/> 邊界中。這將允許其上方的任何 Client Component 都能夠進行靜態渲染。
  • 使用 searchParams Pages prop 將在請求時將該頁面切換為動態渲染。

動態獲取資料​

動態數據提取是指通過將緩存選項設置為 'no-store' 或將重新驗證設置為 0，明確地取消緩存行為的 fetch() 請求。

還可以使用段配置對象設置佈局或頁面中所有 fetch 請求的緩存選項。
