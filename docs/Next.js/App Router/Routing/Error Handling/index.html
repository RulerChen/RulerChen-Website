錯誤處理

error.js 是如何工作的​

error.js 會自動創建一個 React 錯誤邊界，將其包裹在嵌套的子段或 page.js 組件周圍。 從 error.js 文件中導出的 React 組件將用作回退組件（fallback component）。 如果在錯誤邊界內部引發了錯誤，則該錯誤將被捕獲，並渲染回退組件。 當回退錯誤組件處於活動狀態時，錯誤邊界上方的佈局保持其狀態並保持互動，錯誤組件可以顯示用於從錯誤中恢復的功能。

從錯誤中恢復​

有時候錯誤的原因可能是暫時的。 在這些情況下，嘗試再次操作可能會解決問題。

錯誤組件可以使用 reset() 函數，提示用戶嘗試從錯誤中恢復。 當執行該函數時，它將嘗試重新渲染錯誤邊界的內容。 如果成功，回退錯誤組件將被重新渲染結果所替換。

app/dashboard/error.tsx
'use client'
 
export default function Error({
  error,
  reset,
}: {
  error: Error
  reset: () => void
}) {
  return (
    <div>
      <h2>Something went wrong!</h2>
      <button onClick={() => reset()}>Try again</button>
    </div>
  )
}

巢狀錯誤邊界​

嵌套組件層級對於跨嵌套路由的 error.js 文件的行為具有影響：

錯誤會冒泡到最近的父級錯誤邊界。 這意味著 error.js 文件將處理其所有嵌套的子段的錯誤。 通過將 error.js 文件放置在路由的不同層級的嵌套文件夾中，可以實現更或更少細粒度的錯誤用戶界面。 錯誤邊界不會處理在同一段落的 layout.js 組件中引發的錯誤，因為錯誤邊界嵌套在該佈局組件內部。

處理佈局中的錯誤​

error.js 邊界不會捕獲在同一段落的 layout.js 或 template.js 組件中引發的錯誤。 這種有意的層級結構在發生錯誤時保持了兄弟路由之間共享的重要用戶界面的可見性和功能性。

要在特定的佈局或模板內處理錯誤，請將 error.js 文件放置在佈局的父段落中。

要在根佈局或模板內處理錯誤，請使用一個名為 global-error.js 的檔案。

處理根佈局的錯誤​

根目錄下的 app/error.js 邊界不會捕獲根目錄下的 app/layout.js 或 app/template.js 組件中引發的錯誤。 要專門處理這些根組件中的錯誤，請使用 app/global-error.js 。

與根 error.js 不同，global-error.js 錯誤邊界包裹整個應用程序，其回退組件在啟用時替換根佈局。 因此，需要注意的是 global-error.js 必須定義自己的 <html> 和 <body> 標籤。

global-error.js 是最粗粒度的錯誤用戶界面，可視為整個應用程序的"萬能"錯誤處理。 由於根組件通常不太動態，且其他 error.js 邊界將捕獲大部分錯誤，因此很少觸發 global-error.js。

即使定義了 global-error.js，還建議定義一個根 error.js，其回退組件將在根佈局內渲染，其中包括全局共享的用戶界面和品牌。

處理伺服器錯誤​

如果在數據獲取過程中或在服務器組件內部引發錯誤，Next.js 將將生成的 error 對象作為 error prop 轉發到最近的 error.js 文件。

在運行 next dev 時，錯誤將從服務器組件序列化並轉發到客戶端的 error.js。 為了確保在生產環境中運行 next start 時的安全性，將一個通用錯誤消息轉發到 error，同時附帶一個 .digest，其中包含錯誤消息的哈希值。 此哈希值可用於對應服務器日誌。
