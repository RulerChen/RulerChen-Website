概述

fetch() API​

新的獲取資料方法構建在本機fetch() 的 Web API 之上並允許在服務器組件中使用async、await。

React 擴展了 fetch 來提供自動刪除重複請求的功能。 Next.js 擴展了fetch，以允許每個請求設置自己的緩存和重新驗證規則。

在伺服器上獲取資料​

我們建議在服務器組件中獲取數據。這使您能夠：

  • 可以直接訪問後端(資料庫)。
  • 防止敏感信息暴露給客戶端。
  • 這減少了客戶端和服務器之間的來回通信，以及客戶端主線程上的工作。
  • 通過單次往返執行多次數據提取，而不是在客戶端上執行多個單獨的請求。
  • 減少客戶端-服務器瀑布。
  • 根據您所在的區域，數據提取也可以在靠近數據來源的地方進行，從而減少延遲並提高性能。

在組件級別中獲取資料​

在 App Router 中，您可以獲取佈局、頁面和組件內的資料。資料獲取也與 Streaming 和 Suspense 兼容。

平行和有序的獲取資料​

通過並行數據獲取，路由中的請求會被急切地發起並同時加載數據。這減少了客戶端-服務器瀑布和加載數據所需的總時間。 通過順序數據獲取，路由中的請求相互依賴並創建瀑布。 在某些情況下，您可能需要這種模式，因為一次提取取決於另一次提取的結果，或者您希望在下一次提取之前滿足某個條件以節省資源。 然而，這種行為也可能是無意的，並導致加載時間更長。

自動刪除 fetch() 的重複請求​

如果您需要在樹中的多個組件中獲取相同的數據（例如當前用戶），Next.js 會自動將具有相同輸入的fetch()請求緩存在臨時緩存中。 GET 此優化可防止在渲染過程中多次獲取相同的數據。

在服務器上，緩存會持續服務器請求的生命週期，直到渲染過程完成。 此優化適用於 fetch 佈局、頁面、服務器組件、generateMetadata 和 generateStaticParams. 這種優化也適用於靜態生成過程。 在客戶端上，緩存會在整個頁面重新加載之前持續會話的持續時間（可能包括多個客戶端重新渲染）。

靜態和動態的資料獲取​

有兩種類型的數據：靜態和動態。

靜態數據是不經常更改的數據。例如，一篇博客文章。 動態數據是經常更改或特定於用戶的數據。例如，購物車列表。

默認情況下，Next.js 自動執行靜態獲取。這意味著數據將在構建時獲取、緩存並在每個請求中重用。作為開發人員，您可以控制靜態數據的緩存和重新驗證方式。

使用靜態數據有兩個好處：

  1. 它通過最大限度地減少發出的請求數量來減少資料庫的負載。
  2. 資料會自動緩存以提高加載性能。

但是，如果您的數據是針對用戶個性化的，或者您希望始終獲取最新數據，則可以將請求標記為動態，並在每個請求上獲取數據而不進行緩存。

快取資料​

緩存是將數據存儲在某個位置（例如 CDN）的過程因此不需要在每個請求時從原始源重新獲取。

Next.js 緩存是一個持久的 HTTP 緩存可以在全球範圍內分發。這意味著緩存可以自動擴展並根據您的平台在多個區域之間共享（例如 Vercel）。

Next.js 擴展選項對象該 fetch() 功能允許服務器上的每個請求設置自己的持久緩存行為。與組件級數據獲取一起，這允許您直接在使用數據的應用程序代碼中配置緩存。

在服務器渲染期間，當 Next.js 遇到提取時，它將檢查緩存以查看數據是否已經可用。如果是，它將返回緩存的數據。如果沒有，它將獲取並存儲數據以供將來的請求。

重新獲取資料​

重新驗證數據 重新驗證是清除緩存並重新獲取最新數據的過程。當您的數據發生更改並且您希望確保應用程序顯示最新版本而無需重建整個應用程序時，這非常有用。

Next.js 提供兩種類型的重新驗證：

  1. 時間：以特定時間間隔重新驗證數據。
  2. 需求：每當有更新時重新驗證數據。

分流和停止​

分流和停止是新的 React 功能，允許您逐步渲染 UI 的渲染單元並將其增量流式傳輸到客戶端。

通過服務器組件和嵌套佈局，您可以立即呈現頁面中不特別需要數據的部分，並顯示正在獲取數據的頁面部分的加載狀態。這意味著用戶不必等待整個頁面加載即可開始與其交互。

舊方法​

新的 App Router 不支持以前的 Next.js 數據獲取方法，例如 getServerSideProps、getStaticProps、getInitialProps ，但是仍然可以在 Pages Router 中使用它們。
